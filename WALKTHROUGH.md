## 1. What Is ReliefSync?

ReliefSync is a **decision-support tool** for disaster relief operations during Philippine typhoons. It answers one specific question:

> **"How much water, rice, and medicine does this evacuation center need over the next 72 hours — and how much do we need to send from the warehouse right now?"**

Without a tool like this, a relief officer would need to:
- Call each evacuation center separately to get stock counts
- Manually calculate how much each person needs
- Guess which camp to prioritize with limited trucks

With ReliefSync, the officer opens a dashboard, selects a camp, clicks one button, and gets an ML-powered forecast with an exact dispatch recommendation in under 2 seconds.

### The Core Workflow

```
Officer selects camp → Clicks "Generate 72-Hour Forecast" → ML model runs
  → Dashboard shows: Current alert level + Depletion chart + Packing manifest
    → Officer reads manifest → Opens Warehouse Kanban → Drags and confirms load
      → Truck is dispatched with exact quantities
```

## 3. How to Run It Locally

> **Before you start:** Make sure Python 3.12 is installed.
> Download it from: https://python.org/downloads/
>
> On this specific machine, always use `py -3.12` instead of `python`
> because there is a conflict with another Python installation (MSYS2).

### Step 1 — Clone the Repository

Open PowerShell and run:

```powershell
git clone https://github.com/myrielleai/ReliefSync.git
cd ReliefSync
```

### Step 2 — Install Python Dependencies

This only needs to be done once, ever.

```powershell
py -3.12 -m pip install -r backend/requirements.txt
```

You should see output ending in something like:
```
Successfully installed flask flask-cors scikit-learn pandas numpy joblib
```

### Step 3 — Train the Machine Learning Model

This only needs to be done once, ever. It creates `model_training/relief_model.pkl`.

```powershell
py -3.12 model_training/train_model.py
```

Expected output:
```
============================================================
  ReliefSync ML Engine — Model Training Initializing...
============================================================
  Dataset created: 900 training records
  Training the Linear Regression model...
  Model trained successfully!

  R² Score:  0.794
  MAE Score: 1216.xx units

  Model saved to: model_training/relief_model.pkl
============================================================
```

> **What does R² = 0.794 mean?** The model explains about 79% of the variation in supply demand. The remaining 21% is noise from random population fluctuations. For a hackathon prototype, this is strong.

### Step 4 — Start the Flask Server

Open a **new PowerShell window** (keep this one open the whole time you're using the app):

```powershell
cd backend
py -3.12 app.py
```

Expected output:
```
============================================================
  ReliefSync Flask API Server Starting...
============================================================
  Server running at: http://0.0.0.0:5000
  Health check:     http://0.0.0.0:5000/api/health
  Predict endpoint: POST http://0.0.0.0:5000/api/predict
  Debug mode: True
============================================================
```

> Keep this window open. If you close it, the ML predictions stop working (the app will use fallback estimates instead).

### Step 5 — Open the Dashboard

**Option A — Simplest (direct file open):**
Navigate to the `frontend` folder and double-click `portal.html`.

**Option B — Local HTTP server (avoids some browser restrictions):**
```powershell
npx serve frontend
```
Then open: http://localhost:3000

### Quick Verification

To confirm everything is connected, open this URL in your browser while the Flask server is running:
```
http://127.0.0.1:5000/api/health
```
You should see:
```json
{
  "status": "ReliefSync API is running!",
  "model_status": "loaded"
}
```

## 5. Understanding the Machine Learning Model

### What Kind of Model Is It?

**Linear Regression** — one of the most well-understood algorithms in machine learning.

### Why Linear Regression (and not something more complex)?

1. **It is interpretable.** You can show a judge the exact equation and explain every number.
2. **It is fast.** Predictions run in less than 1 millisecond. No GPU needed.
3. **It is honest.** Disaster logistics genuinely follows a near-linear relationship: twice as many people needs roughly twice as many supplies.
4. **It is appropriate for the data.** We have 900 training records and 4 features. A neural network would be massive overkill and would likely overfit.

### The Training Data

The model was trained on **900 synthetic records** representing past typhoon responses. "Synthetic" means we generated the data mathematically, but based on real Philippine parameters.

**Each record contains:**

| Column | Type | Range | Description |
|---|---|---|---|
| `camp_id` | Integer | 1–3 | Which evacuation center |
| `supply_category_id` | Integer | 1=Water, 2=Rice, 3=Medicine | What type of supply |
| `current_population` | Integer | 300–3,000 | How many evacuees |
| `pagasa_signal` | Integer | 1–5 | PAGASA typhoon signal level |
| `units_consumed` | Integer | varies | **TARGET** — total units needed in 72h |

**How the target was calculated (the "ground truth" the model learned from):**

```
Water formula:   population × 3.0 × (1 + pagasa_signal × 0.15)
Rice formula:    population × 1.0 × (1 + pagasa_signal × 0.05)
Medicine formula: population × 0.15 × (1 + pagasa_signal × 0.25)
```

These formulas are grounded in:
- **SPHERE Humanitarian Standards:** 15 liters water/person/day, ~400g rice/person/day
- **NDRRMC typicaly operations:** Medical kits cover roughly 15% of evacuees per event
- **PAGASA Signal multiplier:** Higher signals = worse conditions = higher consumption

### The Model Formula (What It Actually Learned)

The trained model learned these approximate coefficients:

```
predicted_demand = -1,846.6 × camp_id
                 + -1,846.6 × supply_category_id
                 +     1.84 × population
                 +   308.4  × pagasa_signal
                 + 6,405.0  (bias/intercept)
```

**Plain English interpretation:**
- For every 1 additional evacuee, demand increases by **1.84 units**
- For every 1 level increase in PAGASA Signal (e.g., Signal 3 → Signal 4), demand increases by **308 additional units**
- The camp_id and supply_category_id coefficients are negative because the model is adjusting for the different scales between categories (water needs vs medical kits are very different magnitudes)
- The bias of 6,405 is the baseline demand before any inputs

### Model Performance

- **R² Score: 0.794** — The model explains ~79% of demand variation. The remaining 21% is random noise from population fluctuations during data generation.
- **MAE: ~1,216 units** — On average, the prediction is within ±1,216 units of the actual need. For dispatch planning, this is an acceptable margin.

### The SPHERE Minimum Floor

After the model makes a prediction, the Flask server applies a minimum floor based on SPHERE humanitarian standards. This prevents the model from returning unrealistically low numbers for any single item:

```python
sphere_floors = {
    "water":   population × 2.0,   # at least 2 units per person
    "rice":    population × 0.5,   # at least 0.5 packs per person
    "medical": population × 0.05   # at least 5% of population (1 in 20)
}
final_dispatch = max(model_prediction, sphere_floor)
```

### The Fallback Mode (When Flask Server Is Offline)

If the Flask server is not running, `portal.js` automatically detects the failed network request and computes estimates using simple multiplication:

```
Water:   population × 2.5
Rice:    population × 0.8
Medical: population × 0.12
```

The dashboard renders normally — the officer never sees an error message. The only difference is the numbers are slightly less precise.

## 7. What to Say to Judges

### The Elevator Pitch (30 seconds)

> "ReliefSync is an ML-powered dashboard for DSWD and LGU disaster officers. During a typhoon, it takes a camp's population, their current stock levels, and the PAGASA signal number — sends it to a Python ML server — and returns the exact quantities of water, rice, and medicine to dispatch within 72 hours. It tells you which camp is most urgent and exactly what to load on the truck. Instead of guessing, relief officers get a data-driven decision in under 2 seconds."

### If Asked "Why Machine Learning?"

> "The demand for relief supplies scales with both population and storm severity in a way that's genuinely predictable from past data. Linear Regression is the right tool here because the relationship is approximately linear: twice as many people means roughly twice as many supplies, and each PAGASA signal level adds a quantifiable number of additional units. We chose a model we can explain fully rather than a black-box neural network that no one — including us — can explain to an LGU coordinator in the field."

### If Asked "Where is the Dataset? How do we see and upload new data?"

> "Click **'View Dataset'** right on the dashboard. It opens our **Dataset Management Center**.
> 1. You can browse the active database records directly in the **Dataset Viewer** tab.
> 2. You can download the active database CSV by clicking **'Download Active CSV Dataset'** in the **Upload & Retrain** tab.
> 3. To upload new data and retrain the model: drag-and-drop a new CSV file into the upload zone. ## 8. Scope and Limitations

### What ReliefSync Does (Scope)

| In Scope | Description |
|---|---|
| Demand forecasting | Predicts 72-hour water, rice, and medical kit needs per camp |
| Alert classification | Classifies each camp as CRITICAL, WARNING, or STABLE based on stock vs. demand |
| Dispatch recommendation | Calculates exact units to send from warehouse |
| Stock depletion visualization | Shows how supplies run out over 72 hours under typhoon conditions |
| Dispatch execution tracking | Kanban board tracks the physical loading of supplies |
| Transparency | ML Insights and View Dataset modals explain every number to non-technical users |
| Fail-safe operation | App works even when the ML server is offline |
| Philippine context | PAGASA Signal 1–5 is the storm severity input, not generic weather data |

---

### What ReliefSync Does Not Do (Limitations)

#### Data Limitations

- **Synthetic training data** — The ML model was trained on generated data, not real historical typhoon incident records. Predictions are illustrative of the model's logic, not validated against real-world outcomes.
- **Static camp database** — The three evacuation centers (Brgy. 172, 173, 174) are hardcoded. Real deployment would need a database of actual LGU-registered evacuation centers.
- **No real-time stock updates** — Stock levels are hardcoded per camp. In production, this would sync with an actual warehouse inventory system.
- **No historical consumption logs** — The chart shows a mathematically generated S-curve, not actual past consumption data from those specific camps.

#### Technical Limitations

- **Backend requires local Python** — The ML Flask server must run on a laptop or cloud server. Vercel (where the frontend is hosted) cannot run Python.
- **No user authentication** — The login screen accepts any credentials. There is no actual security layer.
- **Single-user interface** — The system does not support multiple officers using it simultaneously with synchronized data.
- **No database persistence** — Dispatch records exist only within the current browser session. Closing the browser loses all session data. There is no backend database.
- **Render free-tier cold start** — If the backend is deployed to Render's free tier, there is a ~30-second startup delay after 15 minutes of inactivity.

#### Operational Limitations

- **72-hour window only** — Forecasts are fixed at 72 hours. There is no 24-hour or 7-day view.
- **Three supply categories only** — Water, rice, and medical kits. Other critical relief items (blankets, hygiene kits, power banks, infant formula) are not modeled.
- **Three camps only** — The demo is locked to three pre-configured camps in Caloocan City. Expanding to other LGUs would require code changes.
- **Typhoon events only** — The model's PAGASA Signal input is specific to typhoons. Earthquake, flood, or landslide logistics are not modeled.
- **No road condition modeling** — "Optimal Dispatch Window" is a hardcoded estimate per camp, not a real routing or road accessibility calculation.
- **No inventory replenishment tracking** — Once a truck is dispatched in the Kanban board, the system does not update the camp's stock levels for the next forecast.

---

### What Would a Production Version Need?

1. **Real data pipeline** — Integration with NDRRMC SitRep databases and LGU stock management systems
2. **Real-time stock sync** — Barcode scanning or digital inventory updates at evacuation centers
3. **Multi-user access with role-based permissions** — Separate logins for camp managers, district coordinators, and DSWD officers
4. **Route optimization** — Integration with LGU road condition data to compute actual dispatch windows
5. **More supply categories** — Blankets, hygiene kits, infant formula, cooking fuel
6. **Multi-event support** — Simultaneous tracking of multiple active disaster events
7. **Historical model validation** — Retraining on real NDRRMC historical data and validating predictions against past events before deployment
8. **Offline-first capability** — Progressive Web App (PWA) that works without internet since evacuation centers may have poor connectivity
